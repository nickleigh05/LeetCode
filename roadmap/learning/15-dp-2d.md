*Same engine, two indices: grids and sequence pairs.*

Two-dimensional DP is the exact same machine as [1-D DP](14-dp-1d.md) with a second index: grid paths, edit distance, longest common subsequence, knapsack. The state is now a pair `(i, j)`, usually filled as a table. If 1-D DP feels solid, the jump is mostly bookkeeping — read the template, then drill the grid and two-sequence problems.

## Concept

2-D DP is [1-D DP](14-dp-1d.md) with **two** indices in the state. Instead of `dp[i]` you keep
`dp[i][j]`, a table you fill cell by cell. Two shapes cover almost everything:

### Grid DP — paths through a matrix

*How many unique paths from the top-left to the bottom-right, moving only right or down?*
Each cell is reachable only from the cell **above** or the one to its **left**, so:

```
dp[i][j] = dp[i-1][j] + dp[i][j-1]      (top row & left column are all 1)

   1   1   1   1
   1   2   3   4
   1   3   6  10        ← dp[2][3] = 6 + 4 = 10 paths to here
   1   4  10  20        ← answer: bottom-right
```

The whole grid fills in O(m·n). Edit distance, min-path-sum, and "can I carve this grid" problems
are the same idea with a different transition.

### Two-sequence DP — comparing strings

*Longest Common Subsequence of `"ABCD"` and `"ACDF"`?* Let `dp[i][j]` = LCS of the first `i`
chars of one string and first `j` of the other:

```
dp[i][j] = dp[i-1][j-1] + 1                 if chars match
         = max(dp[i-1][j], dp[i][j-1])      otherwise

        ""  A  C  D  F
    ""   0  0  0  0  0
    A    0  1  1  1  1
    B    0  1  1  1  1
    C    0  1  2  2  2
    D    0  1  2  3  3   ← LCS length = 3  ("ACD")
```

The recipe never changes: **name the state `(i, j)`, write the transition, seed the base
row/column.** If 1-D DP feels solid, 2-D is mostly careful bookkeeping over a grid.

### Interval DP — optimal answer over a subrange

Grid DP and two-sequence DP both use `(i, j)` as row and column indices. Interval DP is different: `dp[i][j]` is the optimal answer for the **subarray/substring from index i to j**. This gives it its own distinct feel — the state is a *range*, not a position pair.

**Recognition signal:** "Optimally combine/reduce a contiguous subrange of the input." — Burst Balloons, Stone Merge, Palindrome Partitioning, Matrix Chain Multiplication.

```
  Minimum cost to merge contiguous stone piles:
  stones = [3, 5, 1, 2]

  dp[i][j] = minimum cost to merge stones[i..j] into one pile.
  cost of any merge = sum(stones[i..j])  (you pick up the total weight)

  Base case: dp[i][i] = 0  (one pile, already merged)

  Fill by increasing range length (len 2, then 3, then 4):

  len=2:
    dp[0][1] = dp[0][0]+dp[1][1] + sum(0,1) = 0+0+8 = 8
    dp[1][2] = 0+0+6 = 6
    dp[2][3] = 0+0+3 = 3

  len=3:
    dp[0][2]: split at k=0: dp[0][0]+dp[1][2]+sum(0,2)= 0+6+9 = 15
              split at k=1: dp[0][1]+dp[2][2]+sum(0,2)= 8+0+9 = 17
              dp[0][2] = min(15,17) = 15
    dp[1][3]: split k=1: 0+3+8=11, k=2: 6+0+8=14 → dp[1][3]=11

  len=4:
    dp[0][3]: split k=0: 0+11+11=22
              split k=1: 8+3+11=22
              split k=2: 15+0+11=26
              dp[0][3] = 22
```

```python
def min_cost_merge(stones):
    n = len(stones)
    prefix = [0] * (n + 1)
    for i in range(n):
        prefix[i+1] = prefix[i] + stones[i]

    dp = [[0] * n for _ in range(n)]

    for length in range(2, n + 1):          # iterate range lengths
        for i in range(n - length + 1):
            j = i + length - 1
            dp[i][j] = float('inf')
            range_sum = prefix[j+1] - prefix[i]
            for k in range(i, j):           # try every split point
                dp[i][j] = min(dp[i][j], dp[i][k] + dp[k+1][j] + range_sum)

    return dp[0][n-1]
```

**The template shape — always the same three nested loops:**

```python
for length in range(2, n+1):           # 1. iterate by range length (bottom-up)
    for i in range(n - length + 1):    # 2. iterate left endpoint
        j = i + length - 1             #    right endpoint is determined
        for k in range(i, j):          # 3. try every split point
            dp[i][j] = min/max(dp[i][j], dp[i][k] + dp[k+1][j] + cost(i,j))
```

**Complexity:** O(n³) time, O(n²) space — three nested loops over n elements.

**Key difference from grid/sequence DP:** you must fill the table in order of *increasing range length*, not row by row, because `dp[i][j]` depends on shorter ranges `dp[i][k]` and `dp[k+1][j]`.
